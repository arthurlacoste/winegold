Issue #8 — Expression-based triggers

Date: 2026-07-11. Context: replace extension-only matching with typed, local, deterministic expressions shared by YAML, Settings, and runtime.

What changed

  • Typed recursive AST, lexer, parser, serializer, validator, and evaluator.
  • Nested and, or, and not; string, collection, numeric, regex, presence, and case-sensitive operators.
  • Dragged file, directory, raw URL, and plain text contexts with metadata and URL components.
  • 1 MiB safe limit for local UTF-8 inside; no network content fetch.
  • SQLite trigger_expression migration while preserving legacy extension data.
  • Recursive Settings builder, generated preview, direct editing, and visible syntax/type errors.
  • Expression-only new YAML exports and legacy fileExtension import normalization.

Why

One lightweight condition remains easy to configure. Advanced actions can now match nested combinations without executing arbitrary Swift or shell during matching.

Risks and useful context

  • Multi-item drags retain existing semantics: every item must match an action.
  • Legacy actions continue matching accepted_extensions until saved/exported with an expression.
  • Raw URL/text payloads still use local content-addressed files internally so the existing execution flow stays compatible.
  • Unavailable trigger values return false. Unknown execution placeholders remain visible instead of becoming empty silently.
